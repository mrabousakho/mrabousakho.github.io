The methodology behind the 18/18 clear

A deeper walkthrough of method, not just findings — how the hunt was actually worked, including the dead ends.

> clinic --case northpeak_descent --show methodology

Triage first. Before touching a single flag, the alert queue was scoped and grouped by host and account — so it wasn't eighteen separate stories, it was one attacker across eighteen checkpoints.

Attempt before answer. Every query in this hunt was written and run from scratch first. Where the trail went cold, that gap was documented as part of the report rather than guessed around.

Result: 18/18 flags, one of four perfect scores on the range, and a repeatable KQL pattern reused across every dual-SIEM case since.

QUERY PATTERNS

What Actually Cleared the Flags_

Scope to the Estate First

Filter
SecurityEvent
| where Computer in ("host1","host2","host3")
| where TimeGenerated between (datetime(2026-06-16 00:00) .. datetime(2026-06-17 00:00))

In a shared workspace, every query starts with an explicit host and time scope, or it returns someone else's estate.

Build One Timeline, Not Many

Sequencing
union SecurityEvent, DeviceProcessEvents
| where Computer in (estateHosts) or DeviceName in (estateHosts)
| order by TimeGenerated asc

Merging Windows and Linux telemetry into a single ordered timeline is what proved the real access sequence, rather than assuming the obvious story.

Valid Account Reuse

T1078
SecurityEvent
| where EventID == 4624
| where LogonType in (3,10)
| summarize FirstSeen=min(TimeGenerated), Count=count() by Account, IpAddress

The clean, valid-account entry point doesn't show up as a spike — it shows up as a single quiet logon worth cross-referencing against the noisy brute-force decoy.

Confirm the Negative

Absence as Evidence
DeviceFileEvents
| where DeviceName == "targetHost"
| where FileName has_any ("cleared","deleted","wiped")
| summarize count()

A zero-row result here is itself a finding — confirming no anti-forensic cleanup occurred, which narrows the attacker's intent.
